\section{Introduction}
\subsection{Motivation}
This project was initially motivated by my lived experience as a \gls{neurodivergent} developer and by personal challenges in using industrial software development tools. In professional settings, my \gls{ASD} was 
often framed as a ``superpower'' due to my ability to rapidly acquire new technical skills and engage in deep logical analysis. However, this perceived strength came with a significant \gls{cognitive_tax}. I found adapting
 to new environments and maintaining a sustainable work-life balance extremely difficult, particularly when faced with task initiation and the decomposition of complex programming problems.
Upon researching existing methodologies and consulting with other neurodivergent developers, it became clear that current Integrated Development Environments (IDEs) offer no effective, ``out-of-the-box'' solutions
 for managing these executive functioning barriers. This gap in assistive technology motivated the development of a solution that transitions the IDE from a passive tool to an active cognitive partner.

Existing \glspl{IDE} can lead to increased cognitive overload and difficulty in maintaining focus. By streamlining interfaces and providing \gls{AI}-driven task decomposition, this project aims 
to reduce the strain on \gls{neurodivergent} individuals in both professional and educational settings, as developers often report challenges with cognitive overload, \gls{hyperfocus}, and breaking
 down more complex problems when programming

\subsection{Problem Statement}
\glspl{IDE} are a primary coding tool for modern software development, yet there are still elements of the interface that remain a significant barrier for \gls{neurodivergent} developers, specifically
 those with \gls{ADHD}, \gls{ASD}, and \gls{dyslexia}. While these tools are feature-rich, they are designed with a ``neurotypical bias'' that assumes a high baseline for filtering distractions and
  managing multi-step planning.

Current IDEs prioritize high feature density, often presenting a constant stream of visual stimuli, including nested menus, status bars, and dense technical jargon. For developers who experience 
sensory overwhelm, this environment creates an immense Extraneous Cognitive Load~\cite{sweller1988cognitive}. This ``visual noise'' competes for limited working memory, often exhausting a developer’s
 ``cognitive budget'' before they can engage with the actual logic of the code.

Research indicates that developers with \gls{ADHD} frequently struggle with task paralysis during the initiation phase of a project~\cite{bramham2009executive}. Furthermore, those with \gls{ASD} may face 
challenges with high-level strategic planning and the decomposition of abstract requirements into actionable steps. Constant notifications—such as linter warnings or compiler recommendations—can inadvertently trigger hyperfocus on peripheral
  details, leading to ``rabbit holes'' that derail the core development goal.

While AI tools are increasingly used in software development, they often focus on generating code rather than supporting the process of being a developer. There is a lack of ``out-of-the-box'' inclusivity in 
these tools, forcing neurodivergent individuals to manually create complex custom configurations to remain productive. Consequently, there is a critical need for an IDE intervention that acts as a 
``digital ramp'', one that filters environmental noise and provides the necessary scaffolding to bypass the freeze response common in task paralysis.

\subsection{Understanding Neurodiversity in Software Development}
Before going into the specific technical barriers of modern \glspl{IDE}, it is necessary to define neurodiversity and the cognitive
 frameworks that underpin this research. Originally coined by Judy Singer in the late 1990s, ``neurodiversity''  views neurological differences—such 
 as \gls{ASD}, \gls{ADHD}, and \gls{dyslexia}, as natural variations in the human genome rather than purely medical deficits~\cite{singer1999why}.

In the context of software engineering, these variations often manifest as unique cognitive profiles. While many \gls{neurodivergent} 
developers possess high aptitude for logic and pattern recognition, they frequently encounter barriers related to Executive Functioning. 
This is an umbrella term for the mental processes that enable us to plan, focus attention, remember instructions, and juggle multiple tasks 
successfully. For a developer, \gls{executive_dysfunction} might manifest as \gls{task_paralysis} when faced with a blank text editor, or an inability 
to filter out the visual noise of a complex interface.

\subsection{Aims and Objectives}

\textbf{Project Aim}

The primary objective of this project is to design, implement, and evaluate an assistive VS Code plug-in that reduces cognitive overload and
 \gls{executive_dysfunction} for \gls{neurodivergent} software developers. By introducing a ``cognitive prosthetic'' layer, the project seeks
  to determine if targeted \gls{UI} simplification and \gls{AI}-driven task decomposition can measurably improve the developer experience by 
  minimising extraneous cognitive load.

\textbf{Research Objectives}

\textbf{Objective 1:} Interface Optimisation (The Minimalist Toggle)

The first aim will be to create a ``Minimalist Toggle''~\allowbreak{}\,using the VS Code Extension to allow users to instantly close all non-essential
 \gls{UI} elements. The toggle hides the Activity Bar, Status Bar, Mini-map, and Breadcrumbs at the same time, reducing the need for extensive menu
  navigation. This addresses the ``Extraneous Cognitive Load'' identified by~\cite{sweller1988cognitive}, reducing visual noise that leads to
   distraction and sensory overwhelm for autistic and \gls{ADHD} developers.

\textbf{Objective 2: }\gls{AI}-Driven Task Decomposition (The Task Architect)

I will develop a \gls{LLM} integration that translates complex, ambiguous programming goals into structured, editable checklists. This element will 
require the implementation of a ``\gls{CoT}'' prompting strategy (\cite{ringel2019ai}) via the Open\gls{AI} \gls{API} that breaks down high-level 
requirements without providing the direct code solution. This supports ``Task Initiation'' and ``Executive Functioning,'' helping users overcome 
the paralysis often associated with large, unstructured projects (\cite{gama2025socio}).

\textbf{Objective 3:} Simplification of Technical Feedback (The Code Explainer)

To build a module that intercepts complex compiler error messages and re-interprets them into plain-language, direct instructions.
Limit output to a maximum of three bullet points using a ``Direct Instruction'' style~\cite{rocha2025conceptual}.
Justification: This minimises the cognitive demand of interpreting dense technical jargon during high-stress moments (such as debugging), 
preventing \gls{hyperfocus} ``rabbit holes.''

\textbf{Objective 4: }Visual Accessibility Customisation (The Visual Styler)

To implement a suite of visual presets focused on reducing eye strain and enhancing readability.
Target: Develop at least two high-contrast and low-stimuli themes that adhere to WCAG 2.1 contrast guidelines~\cite{bengtsson2025web}.
Justification: Addresses the sensory sensitivities and ``attractive colour'' preferences reported by \gls{neurodivergent} developers in recent
 surveys~\cite{rocha2025conceptual}.

\textbf{Objective 5:} Empirical Evaluation of Cognitive Impact

To conduct a comparative study assessing the plugin’s efficacy against a standard \gls{IDE} environment.
Target: Collect and analyse quantitative data via the \gls{NASATLX} and the \gls{SUS}.
Justification: This provides a standardised scientific metric to validate whether the tool successfully reduces
 mental demand and improves the workflow for the target demographic.
